## TL;DR

ENPIRE is a **physical autoresearch** harness: a coding agent runs real robot experiments, reads failure logs, edits policy-learning code, and launches the next experiment. The key move is to make robot policy improvement a repeatable optimization loop: reset, execute, verify, refine.

<Video src="https://research.nvidia.com/labs/gear/enpire/videos/website-teaser.mp4" caption="ENPIRE demo: agentic policy self-improvement running across multiple YAM bimanual robot stations" type="direct" />

## Problem

Dexterous manipulation still depends heavily on human supervision and algorithmic engineering. Data collection, reset after failure, success verification, and training-code adjustment all require human involvement, making real-world policy improvement slow. Coding agents can already automate many digital experiments, but physical robotics lacks the interfaces needed for safe reset, real-world outcome verification, and parallel experimentation across a robot fleet. ENPIRE targets exactly this missing layer.

## Core Idea

ENPIRE decomposes robot policy improvement into four modules that coding agents can operate:

- **Environment (EN)**: builds automatic reset, safety boundaries, reward, and verification APIs from human feedback.
- **Policy Improvement (PI)**: launches policy-improvement regimes such as heuristic learning, behavior cloning, offline/online RL, and tool calling.
- **Rollout (R)**: evaluates policies on one or more physical robots and collects logs.
- **Evolution (E)**: analyzes logs and literature, modifies training infrastructure and algorithm code, and proposes the next experiment.

<Collapsible title="Method Details">

The important design choice is to separate environment construction from policy improvement. In the first phase, a coding agent uses human feedback to build task-specific environment APIs: reset, observation, step, reward, and success verification. Once stabilized, these APIs become immutable interfaces for the policy-improvement phase.

In the second phase, the agent repeatedly improves policies through those APIs. It may tune behavior-cloning regularization, mix online RL with demonstration data, combine perception tool calls with motion planning, or create new experiment branches. With multiple robots, ENPIRE assigns one coding agent per station, tests hypotheses in parallel, and transfers successful ideas through Git and log summaries.

This does not hide physical messiness. Robot dynamics, contact friction, perception failures, and reset variability remain part of the feedback loop, and the agent improves policy recipes against that real-world signal.

## Key Results

- On ENPIRE, frontier coding agents improve policies for real dexterous manipulation tasks including Push-T, pin insertion, GPU insertion, and zip-tie cutting; the abstract reports up to **99% success rate**.
- For pin insertion, agents explore BC, iterative BC, online/offline/offline-to-online RL, and BC regularization while targeting 50 consecutive real-world successes.
- Fleet scaling reduces wall-clock time. In Push-T, scaling from 1 to 8 agents reduces time to normalized score 1.0 from roughly 5 hours to 2 hours; in pin insertion, time to near-perfect success drops from more than 1.5 hours to about 40 minutes.
- Scaling is not free: GPU utilization and token consumption increase, while per-robot utilization decreases.

## Achievements and Limitations

What it newly achieves:

- Ports coding-agent autoresearch from digital ML experiments into real dexterous robot policy improvement.
- Proposes a two-stage structure: first construct environment APIs with automatic reset and verification, then automate policy learning on top of them.
- Parallelizes hypothesis search across robot stations and introduces MRU and MTU as resource-efficiency metrics for physical autoresearch.

Limitations:

- As the authors note, robot and compute resources are not fully saturated. While agents read logs, write code, debug, or wait for the language-model backbone, robots may sit idle.
- Larger fleets reduce wall-clock time but increase token cost super-linearly, buying speed with coordination overhead.
- 🤖 The system assumes a highly integrated robot fleet and internal tool API stack. Reproducing the same level of reset, verification, logging, and sandboxing in another lab will be nontrivial.

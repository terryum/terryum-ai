## TL;DR

에이전트 프레임워크는 모델의 한계에 대한 가정을 인코딩하지만, Claude의 능력이 진화하면 이 가정이 오히려 **성능 병목**이 된다. Anthropic은 범용 도구 활용, 자기 오케스트레이션, 그리고 캐싱·보안 경계를 통한 신중한 제약 설정이라는 3가지 설계 패턴을 제시한다.

## 문제

Chris Olah의 표현대로 Claude 같은 생성형 AI 시스템은 "만들어진다기보다 자란다(grown more than built)." 이는 에이전트 프레임워크 설계에 근본적 도전을 제기한다. 개발자들은 모델의 현재 한계에 맞춰 하네스를 설계하지만, 모델이 빠르게 진화하면서 이전 세대의 가정이 다음 세대의 병목이 된다. 예를 들어, 모든 도구 결과를 컨텍스트 윈도우에 로드하는 패턴은 이전에는 필수였지만 코드 실행이 가능한 모델에게는 불필요한 토큰 낭비다. 이 블로그는 Claude의 진화하는 지능에 적응하면서 레이턴시와 비용을 관리하는 **3가지 설계 패턴**을 제시한다.

## 핵심 아이디어

기존 에이전트 프레임워크가 모델별 특화 도구와 복잡한 오케스트레이션을 구축하는 반면, 이 블로그의 핵심 메시지는 "**Claude가 이미 아는 것을 활용하라**"는 것이다.

- **패턴 1 — 범용 도구 활용**: Claude 3.5 Sonnet이 bash와 텍스트 에디터만으로 SWE-bench Verified 49%(당시 SOTA)를 달성. 특화 도구 대신 Claude가 잘 아는 범용 도구 위에 고수준 기능을 조합
- **패턴 2 — 자율성 확대**: "뭘 그만할 수 있을까?"라는 질문으로 불필요한 구조 제거. 자기 오케스트레이션, 점진적 컨텍스트, 메모리 영속성
- **패턴 3 — 신중한 경계 설정**: 프롬프트 캐싱, 선언적 도구, 보안/UX/관찰가능성 경계

## 구체적 방법

<Collapsible title="구체적 방법">

### 패턴 1: Claude가 아는 것을 활용하라

Claude는 인터넷 학습 데이터에서 bash, 텍스트 에디터 같은 범용 도구를 깊이 이해하고 있다. Claude Code는 이 두 도구만으로 SWE-bench Verified에서 49%를 달성했다(2024년 말 SOTA). 핵심 원칙은 특화 인터페이스 대신 Claude가 이미 잘 아는 도구 위에 고수준 기능을 **조합(composition)**하는 것이다:

- **프로그래밍 도구 호출**: bash를 통한 코드 실행
- **스킬**: 텍스트 에디터로 읽는 태스크별 지침 파일
- **메모리**: 텍스트 에디터로 관리하는 영속적 컨텍스트 파일

### 패턴 2: "뭘 그만할 수 있을까?"

에이전트 하네스의 가정을 지속적으로 검증하고, 불필요해진 구조를 제거해야 한다.

**A. 자기 오케스트레이션**: 기존 패턴은 모든 도구 결과를 Claude의 컨텍스트 윈도우에 로드했다. 그러나 코드 실행 도구(bash/REPL)를 제공하면 Claude가 직접 코드를 작성하여 도구 호출을 연결하고, 필요한 결과만 필터링하여 컨텍스트에 로드할 수 있다. BrowseComp에서 Opus 4.6에 이 능력을 부여하자 정확도가 **45.3%에서 61.6%로 향상**되었다.

**B. 점진적 컨텍스트**: 시스템 프롬프트에 모든 태스크 지침을 미리 로드하면 어텐션 예산이 소진된다. 대신 스킬의 YAML 프론트매터로 간략한 개요만 제공하고, 필요할 때 전체 내용을 읽게 한다. 서브에이전트(격리된 컨텍스트 윈도우)도 활용하여 Opus 4.6는 BrowseComp에서 단일 에이전트 대비 **+2.8%** 향상을 보였다.

**C. 메모리 영속성**: 장기 실행 에이전트는 단일 컨텍스트 윈도우를 초과한다. 해결책은 두 가지다:
- **컴팩션**: Claude가 과거 컨텍스트를 요약하여 연속성 유지. BrowseComp에서 Opus 4.6는 **84%**, Opus 4.5는 68%, Sonnet 4.5는 43%를 달성
- **메모리 폴더**: Claude가 파일에 컨텍스트를 기록하고 필요할 때 읽음. BrowseComp-Plus에서 Sonnet 4.5가 **60.4%에서 67.2%로 향상**

<Figure src="./fig-4.png" caption="Claude가 도구 호출을 직접 코드로 오케스트레이션하여 결과를 컨텍스트에 로드하기 전에 필터링하는 과정." number={4} />

### 패턴 3: 경계를 신중하게 설정하라

**A. 캐시 히트 최대화**: Messages API는 무상태이므로 매 턴마다 컨텍스트를 재패키징해야 한다. 프롬프트 캐싱을 사용하면 캐시된 토큰은 기본 입력 토큰 비용의 **10%**만 소모된다. 원칙: 정적 콘텐츠를 앞에, 동적 콘텐츠를 뒤에 배치하고, 모델을 변경하지 않으며(캐시가 모델별), 도구 정의를 안정적으로 유지한다.

**B. 선언적 도구**: Claude는 애플리케이션의 보안 경계나 UX 표면을 본질적으로 알지 못한다. 다음 경우에 전용 도구를 생성해야 한다:
- 보안 경계가 필요한 액션
- 되돌리기 어려운 액션(외부 API 호출)
- 사용자 확인이 필요한 액션
- 구조화된 로깅/추적/재생이 필요한 액션

</Collapsible>

## 주요 결과

- **SWE-bench Verified**: Claude 3.5 Sonnet이 bash + 텍스트 에디터만으로 **49%** 달성 (2024년 말 SOTA)
- **BrowseComp 자기 오케스트레이션**: Opus 4.6가 도구 출력 필터링 능력으로 **45.3% → 61.6%** (+16.3%p)
- **BrowseComp 컴팩션**: Opus 4.6가 **84%**, Opus 4.5가 68%, Sonnet 4.5가 43%
- **BrowseComp-Plus 메모리 폴더**: Sonnet 4.5가 메모리 폴더 접근으로 **60.4% → 67.2%** (+6.8%p)
- **서브에이전트**: Opus 4.6가 단일 에이전트 대비 **+2.8%** 향상

## 달성점과 한계점

### 새롭게 달성한 점
- 에이전트 하네스 설계를 **모델 진화에 적응하는 3가지 아키텍처 패턴**으로 체계화
- 범용 도구(bash + 텍스트 에디터) 조합만으로 SWE-bench SOTA를 달성할 수 있음을 실증
- 자기 오케스트레이션, 컴팩션, 메모리 폴더 등의 패턴 효과를 BrowseComp 벤치마크에서 정량적으로 검증

### 한계점
- 블로그 형식으로 인해 각 패턴의 적용 조건과 실패 사례에 대한 체계적 분석이 부족하다
- SWE-bench와 BrowseComp에 집중되어 있어 다양한 도메인(고객 서비스, 데이터 분석 등)에서의 일반화 가능성은 검증되지 않았다
- 🤖 프롬프트 캐싱의 10% 비용 절감 외에 3가지 패턴 적용 시의 전체적인 비용-레이턴시 트레이드오프가 정량화되지 않아, 실제 프로덕션 환경에서의 경제성 판단이 어렵다

## TL;DR

LLM 시스템의 성능을 결정하는 것은 모델 가중치만이 아니라 하네스(모델에게 정보를 저장·검색·제시하는 코드)이며, 이 하네스 코드를 에이전트가 자동으로 탐색·최적화하는 Meta-Harness를 제안하여 세 가지 도메인에서 수작업 baseline을 상회한다.

## 문제

LLM 시스템의 성능은 모델 가중치뿐 아니라 **하네스**—모델에게 어떤 정보를 저장하고, 어떻게 검색하며, 어떤 형태로 제시할지를 결정하는 코드—에 크게 좌우된다. 그러나 하네스는 여전히 대부분 수작업으로 설계되며, 기존 텍스트 최적화 도구(OpenEvolve 등)는 피드백을 너무 공격적으로 압축하여 이 세팅에 적합하지 않다. 즉, 최적화 도구가 이전 시도의 "왜 실패했는지"에 대한 풍부한 맥락을 잃어버린다.

## 핵심 아이디어

Meta-Harness는 하네스 코드를 탐색하는 **outer-loop 시스템**이다. 핵심은 기존 텍스트 최적화 도구와 달리, proposer 에이전트에게 **이전 모든 후보의 소스코드, 실행 트레이스, 점수에 대한 전체 접근권**을 파일시스템을 통해 부여하는 것이다:

- **파일시스템 기반 경험 축적**: 각 iteration의 하네스 코드, 추론 트레이스, 평가 점수를 디렉토리 구조로 저장
- **Agentic proposer**: 파일시스템 전체를 읽고, 이전 시도들의 성공/실패 패턴을 분석하여 새 하네스 코드를 제안
- **Harness = LLM + Code**: 하네스가 정보 저장·검색·제시 로직 전체를 포함하므로, 프롬프트 최적화보다 훨씬 넓은 탐색 공간

이 접근법의 핵심 통찰: **피드백을 압축하지 않고 원본 그대로 보존**하면, 에이전트가 "왜 이 전략이 실패했는지"를 추론하고 구조적 변경을 할 수 있다.

### 탐색 루프 구조

각 iteration에서:
1. **Proposer 에이전트**가 파일시스템을 탐색 — 이전 후보들의 코드, 트레이스, 점수 읽기
2. 새 하네스 코드를 Python으로 작성
3. 평가 태스크에서 하네스 실행
4. 결과(코드 + 트레이스 + 점수)를 파일시스템에 저장

### Proposer의 정보 접근

기존 텍스트 최적화 도구와의 결정적 차이:
- **OpenEvolve**: 점수와 짧은 피드백 문자열만 제공
- **TTT-Discover**: 최근 K개 후보의 코드만 제공
- **Meta-Harness**: 모든 이전 후보의 **소스코드 + 실행 트레이스 + 점수** 전체를 파일시스템으로 제공

Table 3의 ablation에서, 소스코드와 점수만 제공해도 기존 방법을 상회하지만, **실행 트레이스**를 추가하면 성능이 추가 향상됨.

### 적용 도메인

1. **온라인 텍스트 분류**: 하네스가 이전 예시를 저장·검색하여 few-shot 컨텍스트 구성
2. **검색 증강 수학 추론**: 하네스가 53.5만 문제 코퍼스에서 관련 문제를 검색하여 컨텍스트 제공
3. **에이전틱 코딩**: 하네스가 에이전트의 도구 사용, 실행 전략, 에러 처리를 결정

## 주요 결과

- **텍스트 분류**: SOTA 컨텍스트 관리(ACE) 대비 **+7.7점**, **4배 적은 컨텍스트 토큰** 사용. OpenEvolve, TTT-Discover를 첫 4회 평가만으로 따라잡고, 최종 **10점 이상 상회**
- **IMO 수준 수학**: 단일 발견된 하네스가 5개 hold-out 모델에서 평균 **+4.7점** (no-retrieval 대비). 가장 강한 고정 baseline(BM25 Retrieval)도 평균적으로 상회
- **TerminalBench-2**: Claude Opus 4.6에서 **47.7% pass rate**로 수작업 baseline(Terminus-KIRA 44.7%) 상회. Haiku 4.5에서 **37.6%**로 모든 Haiku 에이전트 중 1위

## 달성점과 한계점

### 새롭게 달성한 점
- **하네스 자동 최적화의 정립**: 프롬프트 최적화를 넘어, 정보 저장·검색·제시 로직 전체를 자동 탐색하는 패러다임 제안
- **풍부한 피드백의 가치 증명**: 피드백을 압축하지 않고 원본(소스코드 + 트레이스) 그대로 제공하면 구조적 개선이 가능함을 실증
- **모델 간 전이**: 수학 추론에서 하나의 발견된 하네스가 5개 다른 모델에서 일관된 향상을 보임

### 한계점
- **저자 언급**: 탐색 비용이 높아(iteration당 다수의 LLM 호출 + 평가) 소규모 태스크에서는 수작업이 더 효율적일 수 있음. 현재 3개 도메인에서만 검증
- 🤖 파일시스템 기반 경험 축적은 iteration이 증가할수록 proposer의 컨텍스트가 커져, 컨텍스트 윈도우 한계에 도달할 수 있으며, 이에 대한 스케일링 전략이 부재하다
